Replace the `processPhoto` import with `processPhotoOffThread` from `@/lib/photos/photo-processor`. Remove the import of `processPhoto` from `@/lib/photos/process-photo`.

Import `{ toast }` from `sonner`.

Modify the `uploadPhotos` function to implement user-decided behavior:

a) **Sequential processing with counter toast (user decision: one at a time, not parallel):**
   - Before the loop starts, show a toast: `toast.loading('Processing photo 1 of N...', { id: 'photo-processing' })`
   - Inside the loop, update the toast on each iteration: `toast.loading(\`Processing photo ${i + 1} of ${total}...\`, { id: 'photo-processing' })`
   - After loop completes (all photos processed), dismiss the loading toast.

b) **Disable add photo buttons while processing (user decision):**
   - The existing `isUploading` state already disables the camera/gallery buttons. Additionally, also disable them during the `isProcessing` state. Change the `disabled` prop on both Take Photo and Gallery buttons from `disabled={isUploading}` to `disabled={isUploading || isProcessing}`.

c) **Replace `processPhoto(file, options)` calls with `processPhotoOffThread({ file, options })`:**
   - In the offline path: replace `await processPhoto(photo.file, { maxSize: 1920, quality: 0.8, timestamp, gpsCoordinates })` with `await processPhotoOffThread({ file: photo.file, options: { maxSize: 1920, quality: 0.8, timestamp, gpsCoordinates } })`.
   - In the online path: same replacement.

d) **Failure handling (user decision: continue processing rest, report failures at end):**
   - Keep existing behavior where individual failures are caught and counted.
   - After the loop, if failCount > 0 AND successCount > 0, show: `toast.error(\`${failCount} photo(s) failed to process\`)`.
   - If all succeeded: `toast.success('Photos ready', { duration: 3000 })` (auto-dismiss per user decision).

e) **Remove `isProcessing` text from upload button** since toast now handles status. The button can show "Uploading..." without the "Processing..." distinction.

<task type="auto">
  <name>Task 2: Unit tests for photo processor orchestrator</name>
  <files>lib/photos/photo-processor.test.ts</files>
  <action>
Create `lib/photos/photo-processor.test.ts` to test the orchestrator's detection and fallback logic.

**Test approach:** Mock the worker and canvas APIs since jsdom doesn't support them. Focus on orchestration logic, not image processing (that's a Playwright test concern).

**Setup:**
```typescript
import { describe, it, expect, vi, beforeEach, afterEach } from 'vitest';
```

Since the orchestrator uses module-level state (`_supportsOffscreen`, `_worker`, `_workerFailed`), each test needs isolation. Use `vi.resetModules()` in `beforeEach` and dynamically import the module:

```typescript
let processPhotoOffThread: typeof import('./photo-processor').processPhotoOffThread;
let getProcessingStrategy: typeof import('./photo-processor').getProcessingStrategy;

beforeEach(async () => {
  vi.resetModules();
  // Re-import to reset module-level state
  const mod = await import('./photo-processor');
  processPhotoOffThread = mod.processPhotoOffThread;
  getProcessingStrategy = mod.getProcessingStrategy;
});
```

**Mock the core processing module:**
```typescript
vi.mock('./process-photo-core', () => ({
  processPhotoInContext: vi.fn().mockResolvedValue({
    blob: new Blob(['test'], { type: 'image/jpeg' }),
    width: 800,
    height: 600,
    timestamp: '2026-01-01T00:00:00.000Z',
    gpsLat: null,
    gpsLng: null,
  }),
}));
```

**Test cases:**

1. **"falls back to main thread when OffscreenCanvas is not available":**
   - Ensure `OffscreenCanvas` is undefined in the test environment (jsdom default).
   - Call `processPhotoOffThread({ file: createTestFile() })`.
   - Verify `processPhotoInContext` was called with `useOffscreen: false`.
   - Verify it returns a valid ProcessedPhoto.

2. **"reports main-thread strategy when OffscreenCanvas unavailable":**
   - Call `getProcessingStrategy()`.
   - Expect `'main-thread'`.

3. **"returns ProcessedPhoto with correct shape":**
   - Call `processPhotoOffThread({ file: createTestFile() })`.
   - Assert result has: blob (instanceof Blob), width (number), height (number), timestamp (string), gpsLat, gpsLng.

4. **"passes options through to processing core":**
   - Call with specific options: `{ file, options: { maxSize: 1024, quality: 0.5, timestamp: specificDate, gpsCoordinates: { lat: 1, lng: 2 } } }`.
   - Verify `processPhotoInContext` received those options.

5. **"handles processing failure gracefully":**
   - Mock `processPhotoInContext` to reject with an error.
   - Expect `processPhotoOffThread` to reject (caller is responsible for catching per the graceful degradation in core).

**Helper:**
```typescript
function createTestFile(): File {
  const content = new Uint8Array([0xFF, 0xD8, 0xFF, 0xE0]); // JPEG magic bytes
  return new File([content], 'test.jpg', { type: 'image/jpeg' });
}
```

**Note:** Testing actual Worker behavior (message passing, transferables) requires `@vitest/web-worker`. Per research, install it: `npm install -D @vitest/web-worker`. However, if the Worker constructor is not available in jsdom, the tests above still work because they test the fallback path. Add a conditional test group "with worker support" that can be skipped if Worker is not available:

```typescript
describe('with worker support (requires @vitest/web-worker)', () => {
  // These tests verify worker path — skip if Worker not available in test env
  it.todo('routes to worker when OffscreenCanvas is available');
  it.todo('retries on main thread when worker crashes');
  it.todo('times out after 30 seconds');
});
```

Mark these as `.todo` — they document the expected behavior without failing. Full worker integration testing is deferred to Playwright E2E tests where real browser APIs are available.
