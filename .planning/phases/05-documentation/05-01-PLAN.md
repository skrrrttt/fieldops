**1. Overview** (2-3 paragraphs)
- What the offline sync system does (queues mutations when offline, syncs when online)
- The key components: mutation queue (IndexedDB via Dexie), sync processor, batch processor, conflict resolution
- Where code lives: `lib/offline/` for logic, `components/offline/` for UI

**2. Mutation Lifecycle State Machine** (QUAL-02)
- Mermaid `stateDiagram-v2` diagram showing all 5 states and 9 transitions:
  - States: pending, syncing, synced (note: transient), failed, conflict
  - Transitions from research: (new)->pending, pending->syncing, syncing->synced, synced->(deleted), syncing->failed, syncing->conflict, failed->pending, conflict->pending, conflict->(deleted)
- State reference table with columns: State | Description | Duration | Code Location
  - pending: Queued for sync, persists until sync starts, mutation-queue.ts
  - syncing: Being sent to server, during active sync, sync-processor.ts
  - synced: Server accepted (transient - immediately deleted), momentary, sync-processor.ts
  - failed: Network or server error, until retry, sync-processor.ts
  - conflict: Server data differs from expected, until user resolves, sync-processor.ts
- Transition reference table with columns: From | To | Trigger | Code Reference
  - Use the exact 9 transitions from research with code locations

**3. Mutation Types** (brief section)
- Table of 4 mutation types: status, comment, photo, file
- Note which can conflict: ONLY status mutations have conflict detection
- Comment, photo, and file are additive (create-only) so they never conflict

**4. Conflict Resolution Flow** (QUAL-04)
- Mermaid `flowchart TD` diagram showing the conflict detection decision tree:
  - Start: "Sync status mutation"
  - Fetch server state -> error? -> return failed
  - Check: server status_id != previous_status_id? -> No -> Apply update
  - Check: server updated_at > mutation created_at? -> No -> Apply update
  - Check: force_overwrite? -> Yes -> Apply update
  - Otherwise -> Return conflict
- Prose explanation of the 3 conflict conditions (all must be true):
  1. Server's status_id differs from the previous_status_id the client recorded
  2. Server's update happened AFTER the local mutation was queued
  3. The mutation does NOT have force_overwrite flag set
- User resolution options with Mermaid diagram or clear description:
  - "Keep My Change" (local wins): Sets force_overwrite=true, status back to pending, re-queued for sync
  - "Use Server Version" (server wins): Deletes mutation from queue entirely
- Code cross-references to `processStatusMutation()` in sync-processor.ts and `resolveConflict()` in mutation-queue.ts

**5. Processing Architecture**
- Explain that individual mutations follow the state machine above, but processing happens in parallel batches
- Batch processing: data mutations (status, comment) at concurrency 5, file mutations (photo, file) at concurrency 2
- Rate limit handling: exponential backoff with bounded jitter (+/- 25%), max 5 retries
- Data mutations process first, then file mutations
- Reference batch-processor.ts and backoff.ts

**6. Sync Triggers**
- Table of 4 triggers: App comes online (auto-detect), User taps "Sync Now" (manual), Service Worker Background Sync (background), User retries failed (reset + re-sync)
- Code references for each

**7. File Reference**
- Table mapping each source file to its role, for quick navigation
